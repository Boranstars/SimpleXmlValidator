## 构建目标建议

推荐先将核心校验能力编译为静态库，而不是让 GUI 目标直接编译所有校验源文件：

```text
SimpleXmlValidatorCore  # STATIC：core/validation + infrastructure
SimpleXmlValidator      # GUI 可执行程序：gui + main.cpp，链接 Core
SimpleXmlValidatorTests # GTest：链接 Core，不依赖 GUI
```

- `SimpleXmlValidatorCore` 对外暴露 `core/validation` 的公共头文件，内部链接 Xerces-C++ 和 spdlog。
- `SimpleXmlValidator` 只链接 Qt 和 `SimpleXmlValidatorCore`，不得直接链接或包含 Xerces 头文件。
- `SimpleXmlValidatorTests` 链接 `SimpleXmlValidatorCore` 与 GTest，复用同一套生产校验逻辑。
- 当前不建议额外拆出“纯 core 库”和“infrastructure 库”；项目规模较小时会增加 CMake 和依赖管理复杂度。若未来需要替换 Xerces 或提供其他后端，再引入接口库或拆分目标。

#### 运行时设计

```cpp
class XercesRuntime final {
public:
    XercesRuntime();
    ~XercesRuntime();

    bool isReady() const noexcept;
    const std::string& errorMessage() const noexcept;

    XercesRuntime(const XercesRuntime&) = delete;
    XercesRuntime& operator=(const XercesRuntime&) = delete;
};
```

- 进程内只创建一个实例，但不提供可在任意位置调用的公开全局单例。
- 后续由 `main.cpp` 作为组合根持有，生命周期覆盖校验器与解析器。
- 构造时捕获初始化异常并保存错误原因；析构时仅在初始化成功后释放 Xerces。
- 解析器、错误处理器和转码资源必须先销毁，再析构运行时。

#### 错误处理器设计

```cpp
class XercesErrorHandler final : public xercesc::ErrorHandler {
public:
    void warning(const xercesc::SAXParseException& exception) override;
    void error(const xercesc::SAXParseException& exception) override;
    void fatalError(const xercesc::SAXParseException& exception) override;
    void resetErrors() override;

    std::vector<ValidationError> takeErrors();
};
```

- `warning`、`error`、`fatalError` 分别映射为 `Warning`、`Error`、`Fatal`。
- 回调时保存行号、列号和 UTF-8 错误描述。
- `takeErrors()` 使用移动语义交付所有错误
- 不提供内部错误列表引用接口，避免引用生命周期和重置语义歧义。
- XSD 加载阶段可复用该处理器收集诊断，但错误最终分类留给阶段 3。

```cpp
public:

```

#### 校验流程设计

1. `XmlValidator` 分别调用 `InputValidator::check` 检查 XML 与 XSD 路径；任一前置检查失败时立即返回 `Failed`，并在 `message` 中提供可理解原因。
2. 检查 `XercesRuntime` 是否可用；初始化失败时返回 `Failed`，不得继续创建解析器。
3. `XercesSchemaValidator` 配置 Xerces 命名空间与 XML Schema 校验能力，并使用用户选择的、已规范化的绝对 XSD 路径预加载 Schema Grammar。
4. Schema 预加载阶段通过 `LocalResourceResolver` 处理本地 `include`/`import`。XSD 语法错误、依赖缺失、依赖不可读、Schema 加载失败和 Xerces 阻断异常均停止流程并返回 `Failed`。
5. 仅当 XSD 成功加载后，解析 XML 并基于已加载的 Schema Grammar 执行格式良好性和 Schema 校验；第一版不读取 XML 中的远程 `schemaLocation`，也不提供多 Schema 映射配置。
6. XML 解析和校验阶段使用独立或已重置的 `XercesErrorHandler` 收集诊断。只有能够定位到 XML 内容的诊断才转换为 `ValidationError` 并写入 `ValidationResult.errors`。
7. `Warning` 单独出现时不导致校验失败；出现 XML `Error` 或 `Fatal` 时返回 `Invalid`。XML 格式不良、根节点/结构不匹配、必填项缺失、类型或枚举值错误均属于 `Invalid`。
8. 捕获 Xerces 与未预期异常，避免异常穿透调用方；若异常发生在 Schema/引擎阶段返回 `Failed`，若已确认属于 XML 内容问题则保留行列号并返回 `Invalid`。

#### 状态与错误分类

| 场景 | `ValidationResult.status` | `errors` | `message` |
|---|---|---|---|
| XML 格式良好且符合已加载 XSD | `Valid` | 通常为空 | 可为空或提供简短成功说明 |
| XML 格式不良或不符合 XSD | `Invalid` | 保留 XML 诊断的级别、行号、列号和 UTF-8 描述 | 可提供校验失败摘要 |
| XML/XSD 输入检查失败 | `Failed` | 为空 | 说明路径、文件类型、可读性或空文件问题 |
| XSD 语法错误、加载失败或依赖失败 | `Failed` | 为空 | 说明 Schema 文件无效或依赖/路径配置错误 |
| Xerces 初始化失败或内部异常 | `Failed` | 为空 | 说明校验引擎或内部处理失败 |

- XSD 加载阶段收集到的 Xerces 诊断仅用于生成 `Failed.message` 或运行日志，不得填入 XML 错误表格。
- XML 错误应保留发生顺序；行号或列号未知时使用 `0`，不得伪造位置。
- 不通过弱化 XSD 约束、吞掉异常或删除失败样例来获得通过结果。

| 初始状态 | 显示 XML/XSD 文件选择区和开始校验按钮；结果区域隐藏，按钮禁用 | 不显示空表格或空结果提示 |
| 选择 XML/XSD | 显示所选路径；两个路径均存在于界面状态后启用按钮；清理旧结果 | 文件对话框取消时不改变当前状态 |
| 文件拖放 | XML/XSD 各自拥有独立拖放目标，仅接受一个本地文件 | 不以扩展名硬性拒绝文件；实际文件检查仍由 `XmlValidator` 完成 |
| 点击开始校验 | 临时禁用按钮，调用 `XmlValidator::validate`，完成后恢复按钮状态并展示结果 | 第一版保持同步执行，不引入异步、取消或批量校验 |
| 普通校验不通过 | 显示红色提示、错误总数和错误表格 | 不得弹窗打断用户 |
| 阻断失败 | 显示 `ValidationResult::message` 的阻断性弹窗，保留当前路径供重试 | 文件、XSD、引擎和内部异常均按此处理 |

文件对话框提供 XML/XSD 常用过滤器及“所有文件”入口。Qt 路径到 `std::filesystem::path` 的转换必须支持中文和空格路径；GUI 只判断是否已选择路径，路径存在性、普通文件、可读性和空文件判断继续由现有输入检查与 `XmlValidator` 统一处理。

#### 依赖与组合根

1. `main.cpp` 先创建 `LogManager` 和 `XercesRuntime`，其生命周期覆盖 `MainWindow`、`XmlValidator` 和 Qt 事件循环。
2. `MainWindow` 接收必要的项目运行时依赖，内部使用 `XmlValidator`；GUI 源文件不得包含 Xerces 或 spdlog 头文件。
3. GUI 通过项目 `LogModule` 记录文件选择、发起校验和阻断性弹窗；不得直接调用 spdlog。
4. 根 `CMakeLists.txt` 将 `src/gui/MainWindow.cpp` 与 `src/gui/ValidationResultPresenter.cpp` 加入 `SimpleXmlValidator`；测试目标不链接 Qt。

#### 明确不做

- 日志导出、日志查看器和日志配置界面。
- 异步校验、取消、进度条、批量校验和超大 XML 专项优化。
- XML 文件编辑、错误位置跳转、Schema/命名空间手工配置和额外校验选项。
- 复杂主题、动画、持久化最近文件或其他超出 UI 草案的交互扩展。
